# iofund-iq

Personal AI assistant for getting more value out of an I/O Fund subscription. Priority: maximize the app's value/utility to its users. (The earlier "POC pitchable to the I/O Fund team" framing was scrapped 2026-07-02 — pitchability is no longer a design criterion; guardrails that existed only for pitch posture can be revisited, while liability- and paid-content-grounded guardrails still hold.)

## Stack

- **App:** Next.js 16 App Router · React 19 · TypeScript · pnpm · Tailwind v4 (two-theme warm-dark/-light token system) · `next/font` (Fraunces / Hanken Grotesk / Spline Sans Mono) · `recharts` (portfolio donut)
- **Auth:** Neon Auth (powered by Better Auth) — Google OAuth + email/password. (Google sign-in is temporarily disabled in the UI as of 2026-06-25; see the `lib/auth/` note below.)
- **DB:** Neon Postgres via Vercel Marketplace · Drizzle ORM · `@neondatabase/serverless`
- **LLM:** Vercel AI Gateway — `"anthropic/claude-sonnet-4-6"` for chat, article distillation, and the weekly digest. AI SDK v6 (`streamText` + `stepCountIs(5)`). Chat is multimodal: a user can attach/paste a portfolio screenshot and the model reads it inline (tickers + share counts) to call `analyze_portfolio_gap`. (The standalone `generateObject` vision-extraction pipeline was removed 2026-06-21 — see the portfolio-gap convention below.)
- **Prices:** Yahoo Finance public chart endpoint (no API key, no auth). Called by the `analyze_portfolio_gap` chat tool + the `/portfolio` review UI to compute live weights.
- **Broker:** Robinhood **Agentic Trading MCP** (official hosted server, `agent.robinhood.com/mcp/trading`, launched 2026-05-27, beta) — per-user OAuth 2.1 + PKCE public client, `client_id` obtained via open Dynamic Client Registration (2026-07-02, stored as `ROBINHOOD_CLIENT_ID`). Primary source of the user's own holdings (screenshot-in-chat is the fallback). Read-only by construction: `lib/robinhood/mcp-client.ts` allowlists read tools; order/watchlist mutations are unreachable. Robinhood has no read-only OAuth scope (single `internal` scope) — the allowlist IS our write boundary, plus Robinhood's own: agents can only trade in a separately funded "Agentic account" the user would have to create.
- **Storage:** Hybrid — Postgres for structured rows (trades, article metadata + **full distilled body** (FTS-indexed and rendered live), encrypted IOF credentials, IOF current positions); git for prose (strategy/thesis docs, weekly digests) plus a hand-curated `positions-bootstrap.yaml` snapshot. (Article bodies moved Postgres-only on 2026-06-19 — they used to be committed as `data/articles/*.md`; that dir is gone.)
- **Search:** Postgres `tsvector` full-text search on `articles(title, body)` with GIN index; pgvector RAG deferred to Phase 1.
- **Cron:** GitHub Actions + Python `scripts/` for trade poll (#2), article ingest (#3), weekly digest (#4), portfolio ingest (daily — authoritative positions refresh from IOF's portfolio PDF). Resend live (`onboarding@resend.dev` → the operator's email) — the only outbound email from the system.

```bash
cd chat
pnpm install
npx vercel link                              # one-time: links chat/ to the iofund-iq Vercel project
npx vercel env pull .env.local --environment=production
```

### Running

```bash
cd chat && pnpm dev
```

predev copies `data/*.md` → `chat/_data/` so chat tools can read them. Hits port 3000 or next free.

### DB schema changes

```bash
cd chat
pnpm db:generate                              # diff db/schema.ts → new migration SQL
pnpm db:migrate                               # apply to Neon
```

`tsvector` generated columns and GIN indexes are hand-extensions to the Drizzle-generated migration (Drizzle's schema snapshot doesn't track them, which is fine — Postgres manages the generated column).

Migration `0008` (the `waitlist` table) came in with the `marketing` merge (2026-07-03), so the journal runs `0000`–`0009` with no gap. (Earlier notes about a deliberately skipped `0008` described the pre-merge state and no longer apply.)

### Verification

- `cd chat && pnpm build` — type-check + production build.
- `cd chat && pnpm eval` — eval suite against the chat agent. Asserts on tool-call traces, not free-form prose. Preeval runs copy-data.sh.
- `cd scripts && python -m pytest` — pure-logic unit tests for the Python ingest pipeline (position state machine, portfolio-PDF parser, payload parsing). No network/DB/LLM.
- For UI changes, hit dev server in a browser. Type-check passes but doesn't catch rendering issues.

## Conventions and guardrails

- **IOF subscription content is paid material.** Never reproduce article prose verbatim in code, docs, or chat output. The distilled docs in `data/` are transformative summaries; new article ingestion follows the same rule (system prompt enforces this for the chat agent).
- **All user-facing prose follows `WRITING.md`** (repo root): the voice target + AI-tells checklist for the strategy/thesis pages, digests, distilled summaries, chat replies, UI microcopy, and the LLM prompts that generate them. Key rule: the internal repo name "IOF" never appears in user-facing prose — use "I/O Fund" / "the fund" / "the firm". Tone is set in the prompt, not patched in the output.
- **Trades source of truth is Postgres (`public.trades`)**, not the seed CSV/JSON files in `data/`. Manual corrections to rows are safe and expected — the ingest upsert is `ON CONFLICT (id) DO NOTHING`, so a fixed price is never re-clobbered. (IOF's feed occasionally ships a price typo — a 2026-07-02 audit of all 1,290 priced trades against Yahoo day ranges found ~7; the two high-confidence ones, `iof:1294` SIMO and `iof:974` MARA, are corrected in place.)
- **Positions source of truth is Postgres (`public.positions`)**, kept current by three layers: (1) the trade-poll piggyback in `ingest_trades.py` keeps `status` in lockstep with `trades`; (2) `enrich_positions` (same cron) fills provisional company (Yahoo) + category (LLM) on brand-new tickers within 30 min; (3) `ingest_portfolio.py` (daily) overwrites all three metadata fields — company, category, **`baseline_weight_pct`** — authoritatively from IOF's portfolio table PDF. **`source` column encodes provenance**: `portfolio_pdf:<date>` = authoritative (current primary path); `bootstrap_yaml:<date>` = authoritative manual fallback; `trade_replay` = born from a trade, not yet enriched; `trade_replay+enriched` = provisional (Yahoo company + LLM-guessed category, overwritten by the next portfolio-PDF run). Precedence holds because enrichment only fills NULLs and only upgrades `source` from `trade_replay`, never from `portfolio_pdf`.
- **Personal portfolio holdings: Robinhood sync is primary (2026-07-02), screenshot-in-chat is the fallback.** A connected user's positions live in the `broker_holdings` snapshot (lazy 30-min refresh — nothing polls; consumers trigger refresh on read); `analyze_portfolio_gap` called with no holdings uses it, and explicit screenshot-extracted holdings always win over the sync. For screenshots, the model reads tickers + share counts inline; per-share prices + values are NEVER used from images — weights are always computed live from Yahoo Finance server-side (one weight pipeline for both sources). The chat route injects a per-turn broker-connection note so the model knows which flow applies. (History: the dedicated upload UX was removed 2026-06-21 in favor of in-chat screenshots; `user_holdings` dropped in migration `0007`; broker read returned 2026-07-02 via Robinhood's official Agentic Trading MCP, which removed the TOS/headless blockers that ruled it out originally.)
- **The Robinhood integration is read-only by construction and stays that way.** Write-side (order placement) remains out of scope — the read-tool allowlist in `lib/robinhood/mcp-client.ts` is the enforcement point; never add mutating MCP tools to it or expose them via chat tools/routes. OAuth tokens (not passwords) are AES-encrypted per user under `ROBINHOOD_TOKEN_ENCRYPTION_KEY` — unlike the IOF key this one is rotatable, since users repair a lost token by reconnecting.
- **Two-layer auth:** Neon Auth identifies the app user; `public.iof_credentials` stores their AES-encrypted IOF email+password keyed to `neon_auth.user.id`. **Never put `IO_FUND_USERNAME`/`IO_FUND_PASSWORD` in Vercel env vars.** They live only in GH Actions secrets (for cron jobs that run as the operator) and in encrypted DB rows (for end-user IOF subscriptions).
- **Registration is invite-only.** The Neon Auth `user.before_create` webhook (`/api/webhooks/neon`) rejects any email not in `NEON_AUTH_ALLOWED_EMAILS`, for both Google OAuth and email/password — registered once via the Neon API per branch (re-registration curl + IDs in `thoughts/shared/plans/2026-06-20-production-go-live.md`). **Changing the allowlist requires a redeploy** (env vars apply to new deployments only). Neon Auth *method* config (e.g. enabling email verification) is changeable only via the **Neon Console UI** — the `configure_neon_auth` MCP and REST `PATCH /auth/config` do not work for it.
- **`IOF_CREDS_ENCRYPTION_KEY` is unrotatable** — losing or changing it bricks every stored IOF credential. Rotation requires a re-encrypt migration.
- **Auth → app redirects must be a full document load, not SPA nav.** The auth pages import `@neondatabase/auth-ui/css`, whose un-layered Tailwind Preflight overrides the app's `@layer` utilities if it lingers in the DOM. So `providers.tsx` sets the Neon Auth provider's `navigate`/`replace` to `window.location.assign`/`replace`, not `router.push` — reverting to client-side nav reintroduces a post-sign-in flash of cramped headings + collapsed `md:` nav links until a manual reload.
- **Citations are render-layer, not prompt-driven.** The Sources block in `chat-thread.tsx` is built deterministically from the `read_article` tool-call trace. The model writes free-form prose; the renderer guarantees attribution. Don't add citation rules to the system prompt.
- **Don't auto-commit changes to the strategy/thesis docs** (`io-fund-{strategy,thesis}.md` or their `.agent.md` pairs). The weekly digest opens an auto-PR to a branch (thesis docs only); the maintainer approves manually. The daily article cron writes only to Postgres now — it makes no git commits (as of 2026-06-19; previously it committed `data/articles/*.md` to main).
- **The app stays invite-only.** The public surface is the marketing landing at `/` + the waitlist endpoint (merged 2026-07-03, opened in `proxy.ts`); everything else sits behind the allowlist gate. Widening the public surface is the maintainer's call, not a default.

## End-goal product

Three hero features (Phase 0 target):

1. **Natural-language chat** over IOF subscription content (live as of 2026-05-19; FTS retrieval + render-layer sources as of 2026-05-20).
2. **Weekly auto-digest** of new IOF activity (live as of 2026-05-20; opens auto-PRs updating both thesis docs — agent + human page — when drift detected).
3. **Personal portfolio gap analysis** (live diff vs IOF current book) — primary: connected Robinhood account, synced holdings feed `analyze_portfolio_gap` with no screenshot and power the `/portfolio` Compare view + `get_my_portfolio` / `get_my_realized_pnl` chat tools (live 2026-07-02). Fallback: attach/paste a portfolio image in chat, the model extracts holdings inline (live 2026-06-20). The earlier dedicated screenshot-upload UX (saved to `user_holdings`, live 2026-05-23) was removed 2026-06-21.

Architecture is **multi-tenant-clean from day one** — every per-user table keyed by `user_id`. The Phase 2 multi-tenant rollout adds Postgres RLS policies, not an auth/data-model rewrite.
